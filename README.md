Policy‚Äëfirst dependency and TypeScript hygiene for monorepos ‚Äî with reasons. Every finding explains why the rule applies, so reviews stay focused and predictable. üö•

## What & Why

- What it is: A lightweight, policy‚Äëdriven guardrail for repository‚Äëwide dependency boundaries. It detects boundary crossings and can fail CI. Typical use: enforce public‚ÄëAPI‚Äëonly imports, keep UI/domain layers separate, align peerDependencies with bundler externals, keep tsconfig sane, and govern skipLibCheck ‚Äî always with an explicit ‚ÄúBecause: ‚Ä¶‚Äù.
- Problems it solves: deep import leaks, accidental cross‚Äëpackage coupling in monorepos, type/exports drift that breaks publishing, peer vs bundler external mismatches, JSX option inconsistencies, and more.
- How it compares and when to use which:
  - [ESLint](https://eslint.org): great for per‚Äëfile AST/style rules. Keep ESLint for in‚Äëfile concerns; use dep‚Äëfence for cross‚Äëfile/package dependency edges and architectural layers.
  - [madge](https://github.com/pahen/madge) / [dependency‚Äëcruiser](https://github.com/sverweij/dependency-cruiser): excellent for graph visualization and flexible analysis. Use those for exploration/complex graph rules; use dep‚Äëfence for an opinionated, CI‚Äëfirst, monorepo‚Äëfriendly policy engine with simple allow/forbid semantics.
  - [syncpack](https://github.com/JamieMason/syncpack): keeps versions and workspace ranges tidy in package.json. Use syncpack for manifest hygiene; use dep‚Äëfence for runtime/build‚Äëtime import and peer/bundler alignment.
  - [publint](https://publint.dev): validates the shape of published packages. publint protects consumers; dep‚Äëfence keeps your source respecting boundaries before you publish.

## Why dep‚Äëfence? ‚ú®
- Condition‚Äëdriven rules (e.g., apply only to packages that are UI + publishable).
- Prevent double‚Äëbundling by aligning tsup `external` with `peerDependencies`. üì¶‚ûïüì¶‚ùå
- Govern `skipLibCheck` with explicit reasons or an allow‚Äëlist. ‚öñÔ∏è
- Keep `tsconfig` healthy (baseline inheritance, forbid `../src` direct references, JSX option sanity).
- Every message includes ‚ÄúBecause: ‚Ä¶‚Äù, making policy intent visible. üó£Ô∏è

## Zero‚ÄëConfig Mode üöÄ

You can run dep‚Äëfence with zero configuration in a typical monorepo:

- Auto‚Äëdetects repo root via nearest `pnpm-workspace.yaml` or `package.json`.
- Scans packages under `packages/*/**/package.json` and `app/package.json` (if present).
- Applies sensible default policies that cover UI/peer/tsup alignment, `tsconfig` hygiene, and `skipLibCheck` governance.
- No `dep-fence.config.*` required to get value on day one.

Why this matters (concrete benefits):
- Instant adoption: drop into CI and get actionable findings without setup.
- Predictable baselines: the same defaults across repos reduce bikeshedding.
- Safe by default: read‚Äëonly checks; use `--strict` to fail on ERRORs when ready.
- Incremental rollout: start as a linter, then codify exceptions as you learn.
- Reviewer‚Äëfriendly: every finding says ‚ÄúBecause: ‚Ä¶‚Äù to explain the policy.

## Quick Start üöÄ

### TS (typed, recommended) and MJS (zero‚Äësetup)

After adding a minimal config, run `pnpm dep-fence`. TS first, then MJS.

TS: `dep-fence.config.ts`

```ts
import { defaultPolicies } from 'dep-fence';

  { id: 'ban-deep-imports', when: () => true, because: 'Use public API only; avoid cross‚Äëpackage internals.', rules: [

  ]},

```

```mjs
  ]},

```

Commands and expected output:

## Representative Policy Examples (Purpose / Snippet / Outcome)

- Public API only (ban deep imports across packages)
  - Purpose: `@org/foo` is OK; `@org/foo/src/x` is not.
  - Snippet:
    ```ts
    { id: 'public-api-only', when: () => true, because: 'Only use package public entrypoints.', rules: [
      { rule: 'import-path-ban', options: { forbid: ['^@[^/]+/[^/]+/src/'] }, severity: 'ERROR' },
    ]}
    ```
  - Outcome: Detects cross‚Äëpackage references into `src/`.

- Peers √ó tsup externals alignment
  - Purpose: ensure peers are treated as externals by the bundler.
  - Snippet:
    ```ts
    { id: 'tsup-peer-hygiene', when: isPublishable(), because: "Don't bundle peers.", rules: ['peer-in-external','external-in-deps'] }
    ```
  - Outcome: Flags peers missing in `tsup.external` or duplicated in `dependencies`.

- Enforce types from dist
  - Purpose: published types should come from `dist/*.d.ts`.
  - Snippet:
    ```ts
    { id: 'package-types-dist', when: isPublishable(), because: 'Expose types from dist/*.d.ts.', rules: ['package-types-dist'] }
    ```
  - Outcome: Violates when `types` or `exports[entry].types` do not point to `dist/*.d.ts`.

## What It Detects (Conceptual Summary)

- Terms: Policy (target selection + Because + rules), Rule (individual check), Finding (INFO/WARN/ERROR).
- Flow: infer package attributes ‚Üí choose applicable policies ‚Üí run rules ‚Üí emit findings with Because.

## Best Practices

- Start with ‚Äúban deep imports‚Äù and ‚Äúpeers √ó bundler external alignment‚Äù.
- Keep exceptions justified with Because; introduce via `severityOverride` (WARN ‚Üí ERROR) for gradual rollout.
- Use `--strict` in CI; run without it locally for discovery.

## Advanced Settings

### Config format choice (TS or MJS)

- Supported: `dep-fence.config.ts`, `dep-fence.config.mjs` (ESM `.mjs` preferred over `.js`).
- Recommendation: prefer `.mjs` for zero‚Äësetup CI/offline; prefer `.ts` for editor type‚Äësafety.
- Running `.ts` configs:
  1) Built‚Äëin fallback (no extra deps): strips type‚Äëonly syntax and evaluates at runtime.
  2) Loader (e.g., `tsx`): `NODE_OPTIONS="--loader tsx" pnpm dep-fence`.
  3) Prebuild (e.g., `tsup`): `tsup dep-fence.config.ts --format esm --dts false --out-dir .`.
- Pitfalls: require Node ‚â• 18 with ESM, avoid mixing CJS/ESM, minimize runtime deps in air‚Äëgapped CI.

### Performance and caching
- Prefer fewer, broader rules to many tiny ones.
- In CI, scope checks to changed packages where possible.

### Workspace/subtree overrides
- Typical setup is a single root config; split into modules or swap via `DEP_FENCE_CONFIG` for subtrees/teams if needed.

## Advanced Policy Configuration üõ†Ô∏è

Zero‚Äëconfig works out of the box. When you need more control, provide an explicit policy file (`dep-fence.config.ts` or `dep-fence.config.mjs`) at the repo root to replace the defaults.

Policy file (TypeScript example):

```ts
// dep-fence.config.ts
import { all, isUI, isPublishable } from 'dep-fence/conditions';
import type { Policy } from 'dep-fence/types';

export const policies: Policy[] = [
  {
    id: 'ui-externals-and-peers',
    when: all(isUI(), isPublishable()),
    because: 'UI packages should not bundle React/MUI; rely on peers.',
    rules: ['ui-in-deps', 'ui-missing-peer', 'peer-in-external']
  }
];
```

You can also tweak severity per rule via `severityOverride` and compose complex conditions with `all(...)`, `any(...)`, and `not(...)`. Helpers like `isUI()`, `isPublishable()`, and `usesTsup()` are available from `dep-fence/conditions`.

Repo‚Äëwide operational settings (JSON) can be declared in `dep-fence.config.json`:

```jsonc
{
  "allowSkipLibCheck": [
    "@your/legacy-chart" // temporary exception
  ]
}
```

Per‚Äëpackage justification for `skipLibCheck` can live in each `tsconfig.json`:

```jsonc
{
  "compilerOptions": { "skipLibCheck": true },
  "checkDeps": {
    "allowSkipLibCheck": true,
    "reason": "3rd‚Äëparty types temporary mismatch; scheduled fix"
  }
}
```

Environment variables:
- `DEP_FENCE_CONFIG` ‚Äî absolute/relative path to a policies module (overrides file discovery).
- `DEP_FENCE_REPO_CONFIG` ‚Äî path to a JSON file with repo‚Äëwide settings (overrides `dep-fence.config.json`).

## Troubleshooting

- False positives with path aliases: align dep‚Äëfence‚Äôs resolver with your tsconfig `paths`/bundler aliases.
- Type‚Äëonly imports flagged: adjust rule targets/conditions or use rules that account for types‚Äëonly edges.
- Dynamic imports: dynamic/computed paths are treated conservatively; model critical boundaries with static paths.

## FAQ

- [ESLint](https://eslint.org) or dep‚Äëfence?
  - Both. ESLint covers in‚Äëfile quality; dep‚Äëfence enforces cross‚Äëfile/package boundaries.
- Why not just [dependency‚Äëcruiser](https://github.com/sverweij/dependency-cruiser)?
  - It‚Äôs great for exploration/visualization. dep‚Äëfence focuses on CI‚Äëfirst, opinionated defaults for monorepos with a small set of high‚Äësignal rules.
- How do we roll it out gradually?
  - Start with WARN and one or two forbid rules; raise to ERROR once violations are addressed.
- How to allow a temporary exception?
  - Use a narrowly scoped policy/condition or `severityOverride`, and record a Because reason.
- How to protect publish quality?
  - Pair dep‚Äëfence (boundaries/types path/peer√óbundler) with [publint](https://publint.dev) (package export surface) in CI.

## What it checks üîç

- Peers √ó tsup externals
  - `peer-in-external` (peer missing from `tsup.external`)
  - `external-in-deps` (external also listed in `dependencies`)
- UI package hygiene
  - `ui-in-deps` (React/MUI/Emotion in `dependencies`)
  - `ui-missing-peer` (UI libs used but missing from `peerDependencies`)
- TypeScript hygiene
  - `tsconfig-no-base` (not extending repo base; hint)
  - `paths-direct-src` (direct `../src` references)
  - `jsx-mismatch` (.tsx present but `jsx` not `react-jsx`)
- `skipLibCheck` governance
  - `skipLibCheck-not-allowed` (enabled without permission)
  - `skipLibCheck-no-reason` (permitted but missing rationale)
- Encapsulation rules
  - `maplibre-direct-dep` (direct MapLibre deps outside the wrapper package)

All findings include ‚ÄúBecause: ‚Ä¶‚Äù to surface rationale.

### Default Policies (catalog) üìö

Each default policy explains why it applies (Because) and targets specific attributes:

- `ui-externals-and-peers` ‚Äî UI packages must not bundle React/MUI; rely on peers. Rules: `ui-in-deps`, `ui-missing-peer`, `peer-in-external`.
- `tsup-peer-hygiene` ‚Äî bundlers must externalize peers. Rules: `peer-in-external`, `external-in-deps`.
- `publishable-tsconfig-hygiene` ‚Äî keep published `tsconfig` clean. Rules: `tsconfig-no-base`, `paths-direct-src`.
- `publishable-local-shims` ‚Äî avoid long‚Äëlived local `*.d.ts`. Rule: `local-shims` (default WARN; can be raised).
- `jsx-option-for-tsx` ‚Äî TSX requires `jsx: react-jsx`. Rule: `jsx-mismatch`.
- `skipLibCheck-governance` ‚Äî permissioned toggle with reasons. Rules: `skipLibCheck-*`.
- `non-ui-paths-hygiene` ‚Äî discourage cross‚Äësrc refs broadly. Rule: `paths-direct-src`.
- `maplibre-encapsulation` ‚Äî only wrapper package may depend on MapLibre. Rule: `maplibre-direct-dep`.

Severity override example:

```ts

  {
    id: 'ui-externals-and-peers',
    when: all(isUI(), isPublishable()),
    because: '‚Ä¶',
```

### Opt‚Äëin Extensions (examples) üß™

Additional rules and helpers can be enabled via a policy file:

- `source-import-ban` ‚Äî ban specific named imports from a module. See `examples/policies/source-import-ban/dep-fence.config.ts`.
- `tsconfig-paths` ‚Äî enforce `paths` to point to `dist/*.d.ts` and/or forbid patterns.

### Advanced: Custom Runtime Rules üß©

You can register custom runtime checks directly in a policy (see example below). For configuration format and loader choices, see ‚ÄúAdvanced Settings‚Äù.

```ts
import type { Policy } from 'dep-fence/types';

export const policies: Policy[] = [
  {
    id: 'my-custom-checks',
    when: isPublishable(),
    because: 'repo‚Äëspecific validation',
    rules: [{
      rule: 'custom',
      id: 'check-pkg-field',
      run: (ctx) => {
        const f = [] as any[];
        if (!ctx.pkgJson.customField) {
          f.push({ packageName: ctx.pkgName, packageDir: ctx.pkgDir, rule: 'check-pkg-field', severity: 'WARN', message: 'missing customField', because: ctx.because });
        }
        return f;
      }
    }]
  }
];
```

## How it works üß≠

- Detects repo root via `pnpm-workspace.yaml` or nearest `package.json`.
- Scans package directories:
  - `packages/*/**/package.json`
  - `app/package.json` (if present)
- Infers attributes: `ui`, `publishable/private`, `usesTsup`, `hasTsx`, `browser/node`, `worker`, `next`, `storybook`, `app`.
- Derives `tsup.external` from repo `tsup.base.config.*` and per‚Äëpackage `tsup.config.*` when available.

## Programmatic API üß©

